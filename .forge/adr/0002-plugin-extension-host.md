# 플러그인은 확장 호스트 프로세스에서 실행

WorkDeck에 VSCode식 플러그인 시스템을 도입하기로 하면서(구현은 2차 마일스톤, 아키텍처 반영은 지금부터), 플러그인 실행 모델로 **별도 확장 호스트 프로세스**를 채택한다. 플러그인 코드는 main도 renderer도 아닌 전용 Node 프로세스에서 실행되고, 매니페스트에 선언한 확장 포인트·권한 범위 안에서 공개된 WorkDeck API(IPC)로만 앱과 통신한다. 플러그인 크래시가 앱을 죽이지 못하고, API 계약이 프로세스 경계로 강제되며, 매니페스트 권한 모델과 자연스럽게 결합되기 때문이다. 비용은 구현 복잡도와 IPC 오버헤드.

## Considered Options

- **main 프로세스 내 직접 로드** — 구현이 가장 단순하지만 플러그인 크래시가 곧 앱 다운이고, 프로세스 경계가 없어 API 우회를 막을 수 없어 기각.
- **하이브리드(로직은 호스트, UI는 renderer webview)** — 가장 정교하지만 문서·구현 복잡도가 최대라 v1 모델로는 과해서 기각. UI 기여 방식의 상세는 구현 단계에서 이 ADR을 깨지 않는 범위에서 정한다.
