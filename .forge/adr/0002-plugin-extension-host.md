# 플러그인은 확장 호스트 프로세스에서 실행

WorkDeck에 VSCode식 플러그인 시스템을 도입하기로 하면서(구현은 2차 마일스톤, 아키텍처 반영은 지금부터), 플러그인 실행 모델로 **별도 확장 호스트 프로세스**를 채택한다. 플러그인 코드는 main도 renderer도 아닌 전용 Node 프로세스에서 실행되고, 공개된 WorkDeck API(IPC)로 앱과 통신한다. 프로세스 경계가 **(1) 플러그인 크래시를 앱으로부터 격리**하고 **(2) WorkDeck API 계약을 강제**(플러그인이 코어 내부에 직접 접근하지 못하게 함)하기 때문이다. 비용은 구현 복잡도와 IPC 오버헤드.

**신뢰 모델 정정(2026-07-03)**: 초안은 "매니페스트 권한 모델과 자연스럽게 결합"된다고 썼으나, 이 표현은 오해를 낳는다. 확장 호스트는 OS 샌드박스나 제한 런타임이 아니라 일반 Node 프로세스이므로, 매니페스트의 `permissions` 선언이 플러그인의 OS 자원 접근(`fs`·`net`)을 기계적으로 막지는 못한다. 플러그인은 **설치한 사용자가 완전히 신뢰하는 코드**(VSCode 모델)로 취급하며, `permissions`는 보안 강제가 아니라 설치 시 사용자 동의 화면의 고지용 메타데이터다. 프로세스 경계가 강제하는 것은 *WorkDeck API 표면*뿐이다. OS 자원 수준의 진짜 격리가 필요해지면 이 ADR을 개정해야 한다 — v1(로컬 설치만)에서는 완전 신뢰 모델이면 충분하다는 판단으로 채택하지 않는다. 상세는 [docs/05-plugin-system.md](docs/05-plugin-system.md) 3.1절.

## Considered Options

- **main 프로세스 내 직접 로드** — 구현이 가장 단순하지만 플러그인 크래시가 곧 앱 다운이고, 프로세스 경계가 없어 API 우회를 막을 수 없어 기각.
- **하이브리드(로직은 호스트, UI는 renderer webview)** — 가장 정교하지만 문서·구현 복잡도가 최대라 v1 모델로는 과해서 기각. UI 기여 방식의 상세는 구현 단계에서 이 ADR을 깨지 않는 범위에서 정한다.
